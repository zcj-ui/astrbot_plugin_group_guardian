# 项目记忆

累计反思 20 次

## 遗留问题与待办

1. **重复检测需加强证据链**：必须提供对比摘要或置信度，否则宁可不输出重复判断。
2. **分类与动作对齐**：推荐动作应与最终分类一致；需求已实现时优先 `question`/`documentation`。
3. **标签完整性校验**：基于推荐动作自动映射标签（如"仅改文档"→ `documentation`）；已满足的需求不标高置信度 `enhancement`。
4. **标题优化机会**：原标题含歧义或冗余时主动建议改写；≤5 汉字且无具体对象时必须改写。
5. **优先级结合业务影响**：绕过审核/漏报/提权场景应倾向 `high`，而非仅看技术复杂度。
6. **系统依赖降级策略**：引入外部库需提供 `ImportError` 捕获与降级方案。
7. **可行性方案分层**：提供"最小可行实现"与"高级实现"两阶段选项。
8. **重复检测声明范围**：即使无重复，也需附检索范围或相似度阈值说明。
9. **截图 Issue 响应流程**：信息不足时优先 `question`，凭代码熟悉度枚举可能区域，要求用户文字确认。
10. **协议拼接字段通用剥离**：OneBot 上报字段（comment/message）需先剥离拼接内容再匹配；推动通用函数。
11. **critical 标签阈值从严**：仅安全绕过、全群异常、数据泄露等场景标 0.8+。

## 关键经验

- **复用已有基础设施**（ISSUE27）：直接复用 anti_flood 消息队列做片段拼接检测，避免重复造轮子。
- **风险与缓解措施成对出现**：每个风险附带解决思路，便于快速评估可落地性。
- **需求已满足的分类处理**（ISSUE24/31）：功能已实现时归为 `question`/`documentation`，而非 `enhancement`。
- **短消息拼接检测设计**（ISSUE27）：采用"连续短消息+间隔<5s"；拼接前过滤标点；非文本消息打断窗口。
- **系统依赖敏感性**（ISSUE26）：外部库安装复杂度影响 good first issue 判定，需标注环境要求。
- **Mixin 继承同步维护**（ISSUE18/19）：`Main` 类继承列表遗漏 `CommandsMixin` 导致 6 个核心命令崩溃；MRO 为"左高右低"。
- **OneBot 协议约束**：撤回窗口 2 分钟、禁言最小粒度 60 秒，可行性分析必须引用。
- **踢人入口全量扫描**（ISSUE15）：4 个踢人路径（指令/LLM/远程/审核），需统一重构共享方法。
- **配置设计范本**：`kick_recall_enabled` + `kick_recall_count` 命名规范、类型明确。
- **用户体验优先**：即使技术可行，若与用户直觉冲突（如秒改分），应坚持用户直觉。
- **协议拼接字段剥离**（ISSUE33/41）：审核时剥离 OneBot 拼接内容（引用原文/问题原文），仅匹配用户纯输入；建议形成通用 `strip_protocol_fragments` 函数。
- **标签置信度校准**（ISSUE33）：截图详实、影响面广、根因明确的 Issue，置信度应 ≥0.8。
- **good first issue 门槛把控**（ISSUE35）：避免推荐给多文件修改、涉及配置兼容或权限理解门槛高的问题。
- **语气滤镜 + 复用历史证据**（ISSUE37）：面对情绪化标题先提取纯需求；主动引用 CHANGELOG 历史变更提升可信度。
- **截图 Issue 标准响应**（ISSUE43）：信息不足时分类 `question`，凭代码熟悉度枚举可能区域，要求用户文字确认；语气滤镜剥离情绪。
- **分区块处理多诉求**（ISSUE39）：含多个诉求时明确区分技术栈，分别给建议，避免混为一谈。
- **低质标签过滤规则**（ISSUE39）：噪声标签（置信度<0.2）应配过滤规则剔除。
- **critical 标签从严**（ISSUE41）：仅安全绕过、全群异常、数据泄露等才标 0.8+。
- **需求已满足时 enhancement 降置信度**（ISSUE40）：功能已存在时 `enhancement` 置信度从 0.7 降至 0.5~0.6。

## 仓库特有模式

- 多 Mixin 集中继承（`ModerationMixin`, `AntiFloodMixin` 等），需维护 Mixin 清单。
- 审核流程：AC 自动机初筛 → LLM 二次判断 → 撤回 + 说明。
- SQLite 存储（审核日志 + 外置词库 `lexicon.db`），WebUI 迁移助手。
