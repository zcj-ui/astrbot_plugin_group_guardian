## 6. 经验教训

- **安全类 Issue 优先级应倾向 high**：权限绕过、提权、未授权操作等不应仅凭代码量定级。本次分析虽识别了风险，但定 medium 是核心失误，应成为后续分析的刚性原则。

- **重复检测必须有证据链**：空泛的“可能是 #23 的重复”属于无效输出，宁缺毋滥。

- **标题优化不可省略**：对于含混、过于简短或缺乏操作对象的标题，必须提供建议改写。本次未遵守已积累的经验。

- **远程执行路径的身份与权限校验需细化**：在分析中只写“增加校验”是不够的，需要说明如何获取操作者角色（例如通过 WebUI 用户绑定的 QQ 在群中的角色），否则可行性方案不完整。

- **标签推荐应严格匹配新手友好度**：`good first issue` 应避免推荐给有多文件修改、涉及配置兼容性、权限理解门槛高的问题。

- **分类可更精准**：类似权限不一致的问题，可以考虑 `bug + enhancement` 双分类，或者在分析正文明确说明存在 bug 行为，而 enhancement 是完善方案。
