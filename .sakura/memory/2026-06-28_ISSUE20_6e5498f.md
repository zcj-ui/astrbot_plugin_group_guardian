## 6. 经验教训

1. **历史感知优先**：对于已有回滚记录的改动，分析时应强制检查 commit log，避免提出已被否决的方案。本次分析做到了，值得推广。
2. **分类与动作对齐**：分析结论中的“推荐做的事”应当对应到 Issue 的最终分类。建议在分析系统里增加“建议动作→分类映射”逻辑（如“仅更新文档” → `documentation`）。
3. **重复判断需更严谨**：不确切的重复提示宁愿不做，否则会干扰处理流程。可以引入相似度阈值，低于 0.7 不输出。
4. **标签完整度校验**：仓库可能已有 `documentation`、`duplicate` 等标签，分析工具应基于建议动作补充缺失的标签，避免只依赖用户问题原文。
5. **用户体验优先原则**：当技术可行但与用户直觉冲突时，应以用户直觉为准。本次分析坚持了分钟单位，值得肯定。
6. **细粒度工作项拆分**：涉及 CHANGELOG 的更新应单独描述工作量（1 文件、低风险），而不是与核心功能修改混在一起。有助于清晰分配。
