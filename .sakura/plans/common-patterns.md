# 常见问题模式与应对策略

## 模式一：Mixin 继承遗漏

**现象**：新增方法或 Mixin 后，`Main` 类未更新继承列表，导致 `AttributeError`。
**案例**：ISSUE18/19 — `_extract_at_targets` 方法缺失，6 个核心命令崩溃。
**应对**：
- 每次引入新 Mixin 时设置检查点（code review + CI 导入测试）
- 维护 Mixin 清单文档

## 模式二：绕过审核的拆分攻击

**现象**：用户将违规内容分多条短消息发送，绕过逐条审核。
**案例**：ISSUE27 — 分条发送违规词无法被 AC 自动机匹配。
**应对**：
- 复用 `anti_flood` 消息队列做拼接检测
- 连续短消息 + 间隔 < 5s 窗口
- 默认仅词库初筛，防误伤

## 模式三：用户未发现已有功能

**现象**：用户提出功能建议，但该功能已实现（如 WebUI 规则中心）。
**案例**：ISSUE24/31 — 自定义关键词、权限管理功能已存在。
**应对**：
- 优先归类为 `question`/`documentation`
- 引导用户使用现有功能
- 判断是否需要改进文档可发现性

## 模式四：回归缺陷

**现象**：新提交引入的副作用（如 Mixin 继承链遗漏）。
**案例**：ISSUE18 — commit `51e836c` 引入回归。
**应对**：
- 为每个命令方法添加至少一个集成测试
- CI 纳入自动化导入测试
- 历史回滚记录需在分析中强制检查 commit log

## 模式五：外部库依赖陷阱

**现象**：增强功能依赖系统库（如 zbar 二维码解码）。
**案例**：ISSUE26 — 二维码识别依赖 pyzbar + 系统 zbar。
**应对**：
- 提供 `ImportError` 降级策略 + 警告日志
- 标注多平台安装提示
- 区分"最小可行实现"与"高级实现"两阶段
