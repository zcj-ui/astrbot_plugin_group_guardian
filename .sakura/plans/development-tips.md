# 开发技巧与注意事项

## 一、复用已有基础设施

- 在新增功能前，优先扫描现有模块寻找可复用组件
- **案例**：短消息拼接检测直接复用了 `anti_flood` 的消息队列，而非重写一套
- 常见可复用组件：`anti_flood` 队列、`storage.py` 的 CRUD、`onebot.py` 协议封装、`moderation.py` 的审核管线

## 二、风险与缓解成对提供

- 每个技术风险附带一个解决思路，便于快速评估可落地性
- 示例："误伤风险 → 仅短消息 + 同用户 + 短窗口 + 撤回优先"
- 建议在分析模板中固定"风险 | 缓解措施"对照表

## 三、可行性方案分层

- 提供两阶段选项供开发者选择：

| 层级 | 实现 | 复杂度 |
|------|------|--------|
| 最小可行 | 仅词库初筛 + 撤回 | 低 |
| 高级实现 | LLM 二次判断 + 禁言 | 中-高 |

- 附替代方案对比表格，帮助决策

## 四、标签完整性映射

- 基于推荐动作自动推断标签：
  - "仅改文档" → `documentation`
  - "引导使用现有功能" → `question` + `documentation`
  - "修复代码缺陷" → `bug`
  - "新功能建议" → `enhancement`
- 标签与主分类保持一致

## 五、工作量估算技巧

- 按模块拆分（配置项、核心逻辑、测试、文档）
- 注明外部约束（OneBot 限制、API 频率）
- 示例：踢人撤回功能 = 配置 20 行 + 共享方法 30 行 + 4 入口各 15 行 + WebUI 映射

## 六、Issue 分析流程规范

- **语气滤镜**：遭遇情绪化标题时先提取纯需求，保持客观职业素养
- **复用历史决策证据**：主动检索 CHANGELOG 回滚记录等仓库历史，提升分析可信度
- **分区块处理多诉求**：含多个诉求的 Issue 明确区分技术栈分别给建议
- **文档同步触发**：用户因历史变更记录困惑时检查入口文档是否充分体现变更

## 七、审核类插件特有注意

- 区分"用户实际发送文字"与"协议拼接的引用原文"，避免误审系统附加内容
- 用户可配置项涉及格式风险（如自定义 LLM 提示词破坏 JSON）时保留格式约束或回退机制
- 新增配置项/命令别名时同步更新 README 和 `/help` 输出
