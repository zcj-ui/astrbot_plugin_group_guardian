# 经验教训总结

## 一、Issue 分类与处理

1. **需求已满足的分类**：用户提出需求但代码已实现时，归为 `question`/`documentation`，而非 `enhancement`。先引导用户使用现有功能。
2. **分类与动作一致**：如果推荐动作是"仅改文档"，分类应为 `documentation`，避免混淆。
3. **优先级结合业务影响**：绕过审核/漏报场景应倾向 `high`，不仅看技术复杂度。

## 二、重复检测

1. **证据链优先**：必须提供对比摘要或置信度，宁可不输出也不做模糊判断。
2. **相似度阈值**：低于 0.7 不输出重复结论。

## 三、标题优化

1. 单字标题（如"建议""bug""问题"）必须改写。
2. 推荐 commit 风格：`type: 具体描述`。
3. 改写时保留原作者的核心诉求，不扭曲原始意图。

## 四、Mixin 继承维护

1. 新增 Mixin 后必须同步更新 `Main` 类继承列表。
2. 建议通过自动化导入测试 + CI 拦截遗漏（如 `from main import Main` 后检查所有命令可调用）。
3. MRO 规则：**左高右低**，靠左的类优先级更高。

## 五、系统依赖管理

1. 外部库安装复杂度影响 `good first issue` 判定。
2. 必须标注环境要求及降级策略。
3. 依赖缺失时通过 `ImportError` 捕获 + 警告日志兜底。

## 六、用户体验优先

1. 技术可行但与用户直觉冲突时（如禁言秒 vs 分），坚持用户直觉。
2. 配置项默认值应保守，避免大版本升级后误判。
