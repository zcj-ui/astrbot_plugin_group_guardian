# Issue 分析规范

## 一、分类与动作对齐

- **核心原则**：推荐动作应与最终分类一致。
- 纯文档/配置引导类 Issue 优先归为 `documentation` 或 `question`，而非 `enhancement`
- 需求已实现时，归为 `question`/`documentation`，不标 `enhancement`
- 功能建议 → `enhancement`，缺陷报告 → `bug`，文档不足 → `documentation`

## 二、优先级判定

| 场景 | 优先级 | 说明 |
|------|--------|------|
| 绕过已有关键拦截机制 | `high` | 如短消息拼接绕过审核、二维码逃避检测 |
| 核心功能回归故障 | `high`/`critical` | 影响多个常用命令的回归缺陷 |
| 中等功能增强 | `medium` | 有明确价值但非阻塞 |
| 文档/配置不清晰 | `low`/`medium` | 不影响运行但影响用户体验 |

- 优先级结合**业务影响**判断，而非仅看技术复杂度
- 评估现有替代防护措施后再定级

## 三、重复检测

- **必须提供证据链**：对比摘要、关键词匹配或置信度
- 置信度 < 0.7 时不输出重复判断
- 宁可不标记，也不做无依据的"可能是 #XX 重复"
- 建议附加"依据：在标题/正文中匹配到 XXX 关键词"

## 四、标题优化

- 标题含歧义或冗余时主动提供改写建议
- **≤5 汉字且无具体对象时必须改写**
- 推荐采用 `type: 具体描述` 的 commit 风格
- 示例：`"解禁出现bug"` → `"Main 类未继承 CommandsMixin 导致所有 @功能命令异常"`
