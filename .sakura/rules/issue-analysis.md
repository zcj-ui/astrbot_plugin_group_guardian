# Issue 分析规范

## 一、分类与动作对齐

- **核心原则**：推荐动作应与最终分类一致。
- 纯文档/配置引导类 Issue 优先归为 `documentation` 或 `question`，而非 `enhancement`
- 需求已实现时，归为 `question`/`documentation`，不标 `enhancement`
- 功能建议 → `enhancement`，缺陷报告 → `bug`，文档不足 → `documentation`
- 权限不一致等混合问题可同时标注 `bug` + `enhancement`

## 二、优先级判定

| 场景 | 优先级 | 说明 |
|------|--------|------|
| 绕过已有关键拦截机制 | `high` | 如短消息拼接绕过审核、二维码逃避检测 |
| 核心功能回归故障 | `high`/`critical` | 影响多个常用命令的回归缺陷 |
| 中等功能增强 | `medium` | 有明确价值但非阻塞 |
| 文档/配置不清晰 | `low`/`medium` | 不影响运行但影响用户体验 |

- 优先级结合**业务影响**判断，而非仅看技术复杂度
- 提权/绕过审核场景应倾向 `high`，不因代码量低估安全风险

## 三、标签规范

- **置信度校准**：证据充分（截图详实+根因明确+影响面广）时 ≥0.8
- **`critical` 从严**：仅安全绕过、全群异常、数据泄露等场景标 0.8+
- **`good first issue` 门槛**：避免推荐给多文件修改、涉及配置兼容或权限理解门槛高的问题
- **低质标签过滤**：置信度 < 0.2 的噪声标签应剔除
- **需求已满足时降置信度**：已实现功能的 `enhancement` 置信度从 0.7 降至 0.5~0.6

## 四、重复检测

- **必须提供证据链**：对比摘要、关键词匹配或置信度
- 置信度 < 0.7 时不输出重复判断
- 宁可不标记，也不做无依据的"可能是 #XX 重复"
- **声明检索范围**：即使无重复，也附检索范围或相似度阈值说明

## 五、标题优化

- 标题含歧义或冗余时主动提供改写建议
- **≤5 汉字且无具体对象时必须改写**
- 推荐采用 `type: 具体描述` 的 commit 风格
- 情绪化标题剥离情绪，聚焦技术实质

## 六、可行性分析规范

- **方案分层**：区分"最小可行实现"与"高级实现"，注明工作量估算
- **风险预警结构化**：可配置项格式风险必须保留约束或回退机制，风险与缓解措施成对出现
- **系统依赖降级**：外部库不可用时 `ImportError` 兜底 + 警告日志，注明多平台安装提示
- **复用历史证据**：主动检索 CHANGELOG 回滚记录等仓库历史，提升分析可信度

## 七、协议与兼容性

- **协议拼接脏数据剥离**：OneBot 上报字段（`comment`、引用块）匹配前优先剥离用户纯输入
- **兼容性记录**：不同 OneBot 实现行为差异（如 NapCat 拼入引用原文）须在注释中记录
- **远程执行身份细化**：WebUI 跨群操作须说明如何获取操作者群角色

## 八、截图 Issue 标准流程

1. 信息不足时优先分类 `question`
2. 基于代码库枚举多种可能性
3. 要求用户提供文字确认后再转 `bug`/`enhancement`
4. 语气滤镜剥离情绪，保持客观
